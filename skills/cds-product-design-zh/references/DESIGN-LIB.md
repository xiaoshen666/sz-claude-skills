# CDS设计组件库准备指南

## 前置条件
- 功能架构设计完成
- 了解CDS平台设计规范和标准组件库

## 检查步骤
1. **检查项目根目录是否存在 `.lib.pen` 文件**
2. **检查项目根目录是否存在 `design-libraries-readme.md` 文件**

## 处理流程

### 情况1：两个文件都存在
- 确认设计组件库已准备就绪
- 读取 `design-libraries-readme.md` 了解设计风格
- 进入全局设计检查验收流程

### 情况2：文件不存在，使用CDS默认UI库
- 自动复制 `references/ui/cds-components.lib.pen` 到项目根目录为 `ui/cds-components.lib.pen.lib.pen`
- 自动复制 `references/ui/cds-design-libraries-readme.md` 到项目根目录为 `ui/design-libraries-readme.md`
- 确认CDS标准设计组件库已准备就绪
- 读取CDS设计规范了解默认设计风格
- 询问用户是否使用CDS默认UI库或选择其他方案

### 情况3：用户选择其他方案
询问用户设计风格选择：

**选项A：使用CDS标准设计风格**
- 基于CDS平台标准UI组件库
- 遵循CDS设计规范和视觉标准
- 确保与CDS平台的一致性

**选项B：从参考截图提取设计风格**
- 调用 `/vibelockdsn-design-distillation-zh` skill
- 从用户提供的参考截图中提取设计风格
- 生成符合参考风格的设计组件库

**选项C：AI 自行发挥设计风格**
- 根据产品类型和目标用户选择合适的设计风格
- 自动生成设计组件库和风格说明

## CDS组件库构建规范

### 基础组件
- **按钮组件**：主按钮、次按钮、文字按钮、图标按钮
- **表单组件**：输入框、选择器、日期选择器、上传组件
- **导航组件**：顶部导航、侧边导航、面包屑、分页器
- **数据展示**：表格、卡片、列表、标签页

### 布局组件
- **布局容器**：栅格系统、弹性布局、响应式布局
- **页面框架**：基础页面、带侧边栏页面、全屏页面
- **弹窗组件**：对话框、抽屉、提示框、确认框

### 业务组件
- **数据组件**：数据表格、图表组件、统计卡片
- **流程组件**：步骤条、进度条、时间轴
- **文件组件**：文件上传、文件预览、文件管理

### CDS专用组件
- **CDS头部组件**：包含CDS标识、用户信息、通知等
- **CDS侧边栏组件**：标准导航结构
- **CDS数据表格**：支持排序、筛选、分页
- **CDS表单组件**：符合CDS数据录入规范

# CDS全局设计检查验收流程

## 触发时机
设计组件库准备完成后

## 执行流程

### 步骤1：检查是否存在检查报告
```
IF 设计文件夹/全局设计检查报告.md 存在 THEN
    进入步骤3（读取检查报告）
ELSE
    进入步骤2（召唤测试验收助手）
END IF
```

### 步骤2：召唤测试验收助手
提示用户：
```
全局设计已完成，但尚未进行检查验收。
请先调用 cds-product-test-zh 执行全局设计检查。
是否现在召唤测试验收助手？
```

- **用户确认**：调用 `cds-product-test-zh` 执行全局设计检查，等待检查报告生成后返回步骤1
- **用户拒绝**：等待用户完成检查后继续，更新 dsg-session.md 状态为"全局设计待检查"，退出当前流程

### 步骤3：读取检查报告
读取 `设计文件夹/全局设计检查报告.md` 内容

### 步骤4：处理检查结果

#### 情况A：检查结论为"通过"
```
展示检查报告摘要给用户确认：
- 需求分析检查结果
- 功能架构检查结果
- CDS设计组件库检查结果
- 一致性检查结果

IF 用户确认OK THEN
    在 dsg-session.md 标记"全局设计检查报告"状态为"已完成"
    进入模块设计阶段
ELSE
    根据用户反馈进行调整
    调整完成后交由用户确认
    循环直到用户确认OK
END IF
```

#### 情况B：检查结论为"不通过"
```
展示问题清单和遗漏项清单给用户：
- 需求分析问题
- 功能架构问题
- CDS设计组件库问题
- 一致性问题

根据问题清单进行调整修改
修改完成后交由用户确认：
"已根据检查报告完成以下修改：
 - [修改项1]
 - [修改项2]
 请确认修改是否满足要求？"

IF 用户确认OK THEN
    调用 cds-product-test-zh 重新执行检查
    等待新检查报告生成后，返回步骤1
ELSE
    根据用户指示继续修改
    修改完成后再次交由用户确认
    循环直到用户确认OK
END IF
```

## 检查验收循环处理原则
1. **问题优先级处理**：严重问题必须修复，中等问题建议修复，轻微问题可酌情处理
2. **修改范围控制**：仅修改检查报告中指出的问题，避免引入新问题
3. **用户确认机制**：每次修改后必须交由用户确认，确保修改方向正确
4. **进度同步更新**：用户确认OK后，立即更新 dsg-session.md 状态
5. **重新检查触发**：修改完成后需重新调用tester进行检查，确保问题已解决
