# 软件开发工作流程指南

## 概述
本文档描述了使用Claude Skills进行软件开发的完整工作流程。整个流程分为8个主要阶段，每个阶段都有对应的技能负责。

## 完整工作流程

### 阶段1：需求分析设计
- **技能**：`sz-requirements-analysis-design`
- **输入**：用户需求文档或需求描述
- **输出**：
  - `01需求文档/{项目名称}需求概要文档.md`
  - `01需求文档/{项目名称}需求详细设计文档.md`
- **下一步**：调用`sz-deliverables-review`检查文档质量，然后调用`sz-architecture-design`

### 阶段2：技术架构设计
- **技能**：`sz-architecture-design`
- **输入**：需求详细设计文档
- **输出**：
  - `02技术设计文档/{项目名称}前端技术架构设计文档.md`
  - `02技术设计文档/{项目名称}后端技术架构设计文档.md`
  - `02技术设计文档/{项目名称}整体技术架构设计文档.md`
- **下一步**：调用`sz-deliverables-review`检查文档质量，然后调用`sz-database-api-design`

### 阶段3：数据库和接口设计
- **技能**：`sz-database-api-design`
- **输入**：需求详细设计文档、后端架构设计文档
- **输出**：
  - `02技术设计文档/{项目名称}数据库设计文档.md`
  - `02技术设计文档/{项目名称}ER图文档.md`
  - `02技术设计文档/{项目名称}API接口文档.md`
  - `02技术设计文档/{项目名称}容器化部署文档.md`
  - `02技术设计文档/{项目名称}非容器化部署文档.md`
- **下一步**：调用`sz-deliverables-review`检查文档质量，然后调用`sz-ui-prototype-design`

### 阶段4：UI原型设计
- **技能**：`sz-ui-prototype-design`
- **输入**：需求详细设计文档
- **输出**：
  - `03原型/{项目名称}设计规范.md`
  - `03原型/{项目名称}组件库文档.md`
  - `03原型/{项目名称}原型图文档.md`
- **下一步**：调用`sz-deliverables-review`检查文档质量，然后调用`sz-frontend-implementation`

### 阶段5：前端实现
- **技能**：`sz-frontend-implementation`
- **输入**：UI原型文档、前端架构设计文档
- **输出**：
  - `04frontend/{项目名称}前端项目配置文档.md`
  - `04frontend/`目录下的前端基础结构
  - **自动调用**：`sz-frontend-architect`进行实际代码生成
- **下一步**：调用`sz-deliverables-review`检查实现质量

### 阶段6：后端实现
- **技能**：`sz-backend-implementation`
- **输入**：后端架构设计文档、数据库设计文档、API接口文档
- **输出**：
  - `05backend/{项目名称}后端项目配置文档.md`
  - `05backend/`目录下的后端基础结构
  - **自动调用**：`sz-backend-architect`进行实际代码生成
- **下一步**：调用`sz-deliverables-review`检查实现质量

### 阶段7：集成部署
- **技能**：`sz-integration-deployment`
- **输入**：前后端代码、部署文档
- **输出**：
  - `{项目名称}项目集成文档.md`
  - `{项目名称}本地开发环境搭建指南.md`
  - `{项目名称}部署流程文档.md`
  - 根据部署方式选择生成相应的部署文档
- **下一步**：调用`sz-deliverables-review`检查部署文档质量

### 阶段8：文档检查（贯穿全程）
- **技能**：`sz-deliverables-review`
- **作用**：在每个阶段完成后检查输出物质量
- **检查内容**：完整性、一致性、准确性、规范性、可行性

## 自动化工作流程

### 使用工作流程协调技能
- **技能**：`sz-workflow-coordinator`
- **功能**：自动化协调整个开发流程
- **优势**：
  - 自动调用各个技能
  - 自动触发文档检查
  - 跟踪流程状态
  - 错误处理和恢复

### 使用方式
1. 调用`sz-workflow-coordinator`技能
2. 提供项目名称
3. 选择是否使用Obsidian格式
4. 技能将自动协调整个开发流程

## 部署方式选择

### 在规划阶段选择部署方式
在`sz-implementation-planning`技能中，需要选择部署方式：

1. **纯前端部署**：仅部署前端静态资源
   - 适合简单项目或API已存在的情况
   - 追求部署方便性

2. **前后端分开部署**：前端和后端分别独立部署
   - 适合大型项目或需要独立扩展的场景
   - 追求可维护性和完整性

3. **前后端一起部署**：前后端打包在一起部署
   - 适合小型项目
   - 追求部署方便性

### 选择标准
- **项目复杂度**：简单项目适合一起部署，复杂项目适合分开部署
- **团队规模**：小团队适合一起部署，多团队协作适合分开部署
- **扩展需求**：需要独立扩展前后端时选择分开部署
- **维护性要求**：高维护性要求选择分开部署
- **部署频率**：前后端更新频率不同时选择分开部署

## 使用建议

### 新手用户
1. 从`sz-requirements-analysis-design`开始
2. 按照每个技能的"下一步工作提示"逐步进行
3. 在每个阶段完成后调用`sz-deliverables-review`检查质量

### 进阶用户
1. 使用`sz-workflow-coordinator`自动化整个流程
2. 在规划阶段仔细选择部署方式
3. 根据项目需求调整工作流程

### 团队协作
1. 使用Obsidian格式保存文档，便于知识管理
2. 建立统一的文件夹结构和命名规范
3. 定期进行文档评审和质量检查

## 故障排除

### 常见问题
1. **技能调用失败**：检查技能名称是否正确，文件路径是否存在
2. **文档检查不通过**：根据评审报告修改文档，重新检查
3. **环境配置问题**：检查环境信息是否正确配置
4. **部署失败**：检查部署方式选择是否合理，部署文档是否详细

### 获取帮助
1. 查看每个技能的"注意事项"部分
2. 参考技能的"验收标准"
3. 使用`sz-deliverables-review`技能进行问题诊断

## 进度日志功能

### 功能概述
进度日志功能通过`sz-deliverables-review`技能实现，用于跟踪项目开发进度，支持工作流程的恢复和继续。

### 主要功能
1. **进度跟踪**：记录每个开发环节的完成状态
2. **文档管理**：跟踪所有输出文档的存在状态和最后检查时间
3. **历史记录**：保存完整的工作流程操作历史
4. **恢复支持**：支持项目中断后的恢复和继续

### 日志文件
- **文件位置**：项目根目录下的`{项目名称}-progress-log.json`
- **模板参考**：`/Users/shenzhiqiang/.claude/skills/sz-deliverables-review/progress-log-template.json`
- **更新时机**：每个环节开始、完成、评审通过时自动更新

### 状态定义
- **pending**：等待开始
- **in_progress**：进行中
- **completed**：已完成
- **completed_with_issues**：已完成但有遗留问题
- **failed**：失败
- **paused**：暂停
- **passed**：评审通过
- **passed_with_conditions**：有条件通过

### 恢复策略
1. **完全恢复**：所有前置环节已完成且通过评审，直接进入当前环节
2. **部分恢复**：某些前置环节未完成或未通过评审，需要先完成这些环节
3. **重新开始**：进度日志文件损坏或格式错误，可以重新开始项目

### 使用方式
 1. 每次调用`sz-deliverables-review`技能时，会自动检查进度日志
 2. 如果日志文件不存在，会创建新的进度日志
 3. 如果日志文件存在，会读取当前进度状态
 4. 根据进度状态提供恢复建议和下一步操作指南
