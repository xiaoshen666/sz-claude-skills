# 前端代码生成流程控制

## 概述

本文档定义前端代码生成的完整流程控制机制，支持断点续传和状态跟踪。

> ⚠️ **重要：系统适配要求**
> 
> 生成的前端代码必须适配 SupOS 平台规范，包括：
> - Vite 配置的 base 路径：`/nebule-custom-static/{moduleCode}/`
> - 路由使用 HashRouter
> - API 请求路径：`/msService/{moduleCode}/...`
> 
> **详细系统适配规则请参考**: [FRONTEND-IMPL.md - 系统适配要求](FRONTEND-IMPL.md#系统适配要求重要)

## 使用前提

在生成具体业务代码前，优先结合以下输入一起使用：

- `{功能名称}-需求详细设计.md`
- `{功能名称}-前端架构设计.md`
- `{页面名称}API设计文档.md`
- 用户最终确认的 `moduleCode`
- `.pen` 文件（UI 原型）
- `UI设计规范.md`

## 适用场景

- 根据需求详细设计和 UI 原型生成页面组件
- 根据前端架构设计生成业务组件
- 根据 API 设计文档生成 API 服务层
- 在已有目录结构下快速补齐标准前端代码
- 支持断点续传和状态跟踪

## 启动恢复流程（最高优先级）

每次进入时**必须首先执行**以下流程：

### 步骤0：读取进度文件
```
IF 项目根目录存在 dev-session.md THEN
    读取 dev-session.md
    读取「前端开发进度」部分
    判断当前阶段状态
    执行对应的恢复操作
ELSE
    进入前序产物检查步骤
END IF
```

### 恢复操作决策表
| dev-session.md 前端进度状态 | 当前阶段 | 恢复操作 |
|------------------------|---------|---------|  
| 文件不存在或前端进度未开始 | - | 检查前序产物，从阶段1开始 |
| 阶段1状态为"未开始" | 阶段1 | 从阶段1：类型定义和工具函数生成开始 |
| 阶段1状态为"进行中" | 阶段1 | 继续阶段1的生成工作 |
| 阶段1状态为"已完成"，阶段2"未开始" | 阶段2 | 从阶段2：API 服务层生成开始 |
| 阶段2状态为"进行中" | 阶段2 | 继续阶段2的生成工作 |
| 阶段2状态为"已完成"，阶段3"未开始" | 阶段3 | 从阶段3：业务组件生成开始 |
| 阶段3状态为"进行中" | 阶段3 | 继续阶段3的生成工作 |
| 阶段3状态为"已完成"，阶段4"未开始" | 阶段4 | 从阶段4：页面组件生成开始 |

| 所有已选阶段已完成 | - | 提示用户前端代码生成已完成 |

## 执行流程

### 前置步骤：前序产物检查

在开始前端代码生成前，必须检查以下前序产物是否存在：

```markdown
### 前序产物检查

我已找到以下前序设计产物：
- `{已找到的文件列表}`

当前仍缺少：
- `{缺失的文件列表}`

其中，`{功能名称}-需求详细设计.md` 已存在。请确认是否允许我基于现有文档，并对缺失部分进行合理发挥后继续生成前端代码？
- 若回复 **允许**：我将基于现有产物继续实现
- 若回复 **不允许**：请先补充缺失文档后我再继续
```

**检查规则**：
1. 优先查找 `.pen` 文件、`UI设计规范.md`、`{功能名称}-前端架构设计.md`、`{功能名称}-需求详细设计.md`
2. 最低要求：至少存在 `{功能名称}-需求详细设计.md`
3. 若缺少部分文档，必须征得用户同意后才能继续
4. 检查结果记录到 `dev-session.md（前端开发进度部分）` 中

### 阶段1：类型定义和工具函数生成

**详细指南**: [FRONTEND-TYPES-UTILS-GENERATION.md](FRONTEND-TYPES-UTILS-GENERATION.md)

**执行步骤**：
1. 确认 moduleCode 和功能需求
2. 生成 TypeScript 类型定义文件
3. 生成通用工具函数
4. 更新 dev-session.md（前端开发进度部分）：标记阶段1为"已完成"

**核心交付物**：
- `src/types/{模块名}.ts`
- `src/utils/{功能}.ts`

### 阶段2：API 服务层生成

**详细指南**: [FRONTEND-API-SERVICE-GENERATION.md](FRONTEND-API-SERVICE-GENERATION.md)

**执行步骤**：
1. 确认 API 设计文档
2. 生成 API 模块定义文件
3. 配置请求路径（使用 /msService/{moduleCode}/... 格式）
4. 更新 dev-session.md（前端开发进度部分）：标记阶段2为"已完成"

**核心交付物**：
- `src/api/modules/{模块名}.ts`

### 阶段3：业务组件生成

**详细指南**: [FRONTEND-COMPONENT-GENERATION.md](FRONTEND-COMPONENT-GENERATION.md)

**执行步骤**：
1. 根据前端架构设计识别可复用组件
2. 生成业务组件文件
3. 实现组件逻辑和样式
4. 更新 dev-session.md（前端开发进度部分）：标记阶段3为"已完成"

**核心交付物**：
- `src/components/{组件名}/index.tsx`
- `src/components/{组件名}/index.module.less`

### 阶段4：页面组件生成

**详细指南**: [FRONTEND-PAGE-GENERATION.md](FRONTEND-PAGE-GENERATION.md)

**执行步骤**：
1. 根据 UI 原型和需求文档生成页面组件
2. 集成业务组件和 API 服务
3. 实现页面交互逻辑
4. 更新 dev-session.md（前端开发进度部分）：标记阶段4为"已完成"

**核心交付物**：
- `src/pages/{模块名}/{页面名}.tsx`
- `src/pages/{模块名}/index.module.less`

### 阶段5：路由配置生成（可选）

**详细指南**: [FRONTEND-ROUTE-GENERATION.md](FRONTEND-ROUTE-GENERATION.md)

**前置条件**: 用户选择了"路由配置"

**执行步骤**：
1. 根据页面结构生成路由配置
2. 配置路由懒加载
3. 配置路由守卫（如需要）
4. 更新 dev-session.md（前端开发进度部分）：标记阶段5为"已完成"

**核心交付物**：
- `src/routes/index.tsx`

### 阶段6：国际化资源生成（可选）

**详细指南**: [FRONTEND-I18N-GENERATION.md](FRONTEND-I18N-GENERATION.md)

**前置条件**: 用户选择了"国际化支持"

**执行步骤**：
1. 提取页面中的文案
2. 生成国际化资源文件
3. 更新组件中的文案引用
4. 更新 dev-session.md（前端开发进度部分）：标记阶段6为"已完成"

**核心交付物**：
- `src/locales/zh-CN.ts`
- `src/locales/en-US.ts`

### 阶段7：样式文件生成（可选）

**详细指南**: [FRONTEND-STYLES-GENERATION.md](FRONTEND-STYLES-GENERATION.md)

**前置条件**: 用户选择了"样式文件"

**执行步骤**：
1. 根据 UI 设计规范生成全局样式
2. 生成主题配置文件
3. 更新 dev-session.md（前端开发进度部分）：标记阶段7为"已完成"

**核心交付物**：
- `src/assets/styles/global.less`
- `src/assets/styles/variables.less`

## 阶段执行规范

### 每个阶段执行前
1. **读取跟踪文件**：确认当前阶段和已完成的阶段
2. **加载指南**：读取对应的生成指南文件
3. **确认前置产物**：检查前置阶段的交付物是否存在
4. **确认关键标识**：确认 moduleCode
5. **⚠️ 系统适配检查**：确保生成代码符合以下规范：
   - Vite base 路径：`/nebule-custom-static/{moduleCode}/`
   - 路由使用 HashRouter（不是 BrowserRouter）
   - API 请求路径：`/msService/{moduleCode}/...`
   - CSS 预处理器：使用 LESS

### 每个阶段执行后（必须严格执行 - 强制清理机制）

> ⚠️ **重要：上下文清理是强制执行的，不可跳过！**
> 如果不清理，会导致上下文累积超出模型 102400 token 限制，引发 API 错误。

#### 1. 更新进度文件（dev-session.md）
立即更新 dev-session.md 中的「前端开发进度」部分：
- **阶段状态**：标记对应阶段为"已完成"
- **完成时间**：记录实际完成时间
- **生成文件数**：记录本阶段生成的文件数量
- **下一阶段**：更新下一阶段信息
- **需要读取的指南**：更新指南文件名

#### 2. 记录关键文件路径（在 dev-session.md 中）
在「关键文件路径」部分记录：
- **前序设计文件**：列出需要的文件名称
- **已生成文件**：记录文件类型和数量（不需要完整路径）
- **Git提交**：记录最新commit hash

#### 3. ⚠️ 强制上下文清理（严格执行 CONTEXT-CLEANING.md）

**清理检查清单**（必须逐项确认）：
- [ ] 所有关键结论已写入进度文件（dev-session.md）
- [ ] 前端开发进度已更新（阶段状态表）
- [ ] 所有产物已保存到正确位置
- [ ] 下次任务需要的文件名称已记录（前序设计文件 + 本次生成文件）
- [ ] 恢复上下文的关键信息已记录（下次恢复指南部分）
- [ ] 下一步操作已明确
- [ ] Git 提交记录已更新（最新commit hash）

> 💡 **注意**：Git 提交是开发流程的独立步骤，**不是清理上下文的前置条件**。建议在阶段完成后尽快提交，但如果 token 即将超限，可以先清理上下文再提交。

**关键文件路径记录示例**（必须写入 dev-session.md（前端开发进度部分））：
```markdown
### 下次任务需要的关键文件
- **前序设计文件**：
  - 需求详细设计：`{功能名称}-需求详细设计.md`
  - 前端架构设计：`{功能名称}-前端架构设计.md`
  - API设计文档：`{功能名称}-API设计文档.md`
  - UI原型文件：`{功能名称}.pen`
- **本次已生成文件**：
  - 类型定义：`src/types/{模块名}.ts`
  - API服务：`src/api/modules/{模块名}.ts`
  - 组件文件：`src/components/{组件名}/index.tsx`
- **下次任务起点**：阶段X+1 - {阶段名称}
- **需要读取的指南**：`FRONTEND-{阶段名称}-GENERATION.md`
```

**清理操作**（必须执行）：
1. 向用户展示清理清单（保留内容 vs 清理内容）
2. 获得用户确认后，**必须执行 `/clear` 指令**
3. 清理后，对话历史应压缩为以下格式：

```markdown
### 阶段X完成摘要
- 生成文件：[文件列表]
- 关键决策：[决策记录]
- Git提交：[commit hash]
- 下次任务起点：阶段X+1 - {阶段名称}
- 需要读取文件：[文件清单]
```

**保留内容**（写入进度文件）：
- ✅ 当前开发阶段和步骤
- ✅ 前后端代码目录路径
- ✅ moduleCode
- ✅ Git 提交记录
- ✅ 关键开发决策
- ✅ 用户确认记录

**清理内容**（从对话中移除）：
- ❌ 代码生成过程
- ❌ 编译错误和修复过程
- ❌ 调试信息
- ❌ 中间讨论和试错过程
- ❌ 临时文件路径
- ❌ 详细的代码实现细节

**上下文监控**：
- 如果对话历史超过 30 轮，**必须立即清理**
- 如果接近 token 限制（>80000 tokens），**必须立即清理**
- 不得因任何理由跳过清理步骤

#### 4. Git 提交
完成代码后立即 git 提交：
```bash
git add .
git commit -m "feat: 完成前端阶段X - {阶段名称}"
```

#### 5. 用户确认
展示生成的文件清单，获取用户确认后继续下一阶段：
```markdown
### 阶段X完成确认

已完成以下工作：
- 生成文件：{文件列表}
- 关键决策：{决策记录}
- Git 提交：{commit hash}

请确认是否继续下一阶段？
```

## 断点续传示例

### 场景1：阶段2完成后中断
```
1. 读取 dev-session.md（前端开发进度部分）
2. 发现阶段1和阶段2已完成，阶段3未开始
3. 直接从阶段3：业务组件生成开始
4. 继续执行后续阶段
```

### 场景2：阶段4进行中中断
```
1. 读取 dev-session.md（前端开发进度部分）
2. 发现阶段4状态为"进行中"
3. 检查已生成的页面组件文件
4. 继续完成页面组件的生成
5. 完成后进入阶段5
```

## 注意事项

1. **必须实时更新跟踪文件**：每个阶段完成后立即更新 dev-session.md（前端开发进度部分）
2. **阶段顺序固定**：必须按 1→2→3→4 的顺序执行，5→6→7 为可选阶段
3. **类型安全**：所有阶段都必须使用严格的 TypeScript 类型定义
4. **API 路径规范**：必须使用 `/msService/{moduleCode}/...` 格式，禁止使用 `/inter-api/...` 历史路径
5. **⚠️ 强制上下文清理**：每个阶段完成后**必须**执行 `/clear` 指令清理上下文，不得跳过
6. **上下文监控**：对话超过 30 轮或接近 80000 tokens 时**必须立即清理**
7. **清理检查清单**：清理前必须完成 6 项检查，确保关键信息已保存
