# 模块开发初始化指南

## 概述

本指南定义了模块开发初始化的流程。初始化时会创建/更新 `dev-session.md` 文件，该文件是**唯一的进度跟踪文件**，用于跟踪模块开发进度、记录关键信息和协调前后端开发工作。

## 初始化流程

### 步骤1：创建/更新 dev-session.md

在项目根目录或模块目录下创建或更新 `dev-session.md` 文件，用于记录当前模块的开发任务和进度。

### 步骤2：填充基本信息

根据当前模块的实际情况，填充以下基本信息：
- 功能名称
- moduleCode
- acronym
- 会话开始时间

### 步骤3：确认前后端目录位置

根据模块代码目录查找规则，确认并记录前后端代码目录位置：
- 前端目录：`{moduleCode}-frontend/`
- 后端目录：`{moduleCode}-custom/`

### 步骤4：初始化开发方案选择

让用户选择开发方案：
- 选项A：前端开发（步骤1+2）
- 选项B：后端开发（步骤3+4+5）
- 选项C：模块集成测试（步骤6）

### 步骤5：初始化进度表格

根据开发流程框架，初始化前端和后端开发进度表格，包含所有开发阶段。

## dev-session.md 格式规范

## 字段说明

### 会话信息

| 字段 | 说明 | 示例 |
|------|------|------|
| 功能名称 | 功能模块的中文名称 | 用户管理、订单管理 |
| moduleCode | 模块编码，从 metadata 文件提取 | usermgmt、ordermgmt |
| acronym | 模块缩略码，用于数据库表前缀 | USR、ORD |
| 会话开始时间 | 会话创建时间 | YYYY-MM-DD HH:mm:ss |
| 最后更新时间 | 最后一次更新时间 | YYYY-MM-DD HH:mm:ss |

### 代码目录信息

| 字段 | 说明 | 示例 |
|------|------|------|
| 前端目录 | 前端项目相对路径 | usermgmt-frontend/ |
| 后端目录 | 后端自定义代码相对路径 | usermgmt/usermgmt-custom/ |

### 开发进度

| 字段 | 说明 | 可选值 |
|------|------|--------|
| 状态 | 阶段当前状态 | ⬜ 未开始、🔄 进行中、✅ 已完成、⬜ 未选择 |
| 完成时间 | 阶段完成时间 | YYYY-MM-DD HH:mm:ss |
| 生成文件数 | 该阶段生成的文件数量 | 数字 |

## 使用规范

### 创建时机

`dev-session.md` 文件应在以下时机创建：
1. 开始新模块开发时
2. 从设计阶段进入开发阶段时
3. 用户明确指定要开发的模块时

### 更新时机

1. **阶段完成时**：每个开发阶段完成后更新对应进度
2. **关键决策时**：记录影响后续开发的决策
3. **Git提交时**：更新最新提交信息

### 清理上下文时

1. 确保 dev-session.md 已更新
2. 确保「下次恢复指南」部分完整
3. 执行 `/clear` 指令
4. 下次从 dev-session.md 的「下次恢复指南」快速恢复

### 字段精简原则

#### ✅ 必须保留
1. **功能标识**：功能名称、moduleCode、acronym
2. **进度信息**：已完成阶段、下一阶段
3. **恢复信息**：需要读取的指南、前置产物
4. **目录信息**：前后端代码目录路径
5. **Git记录**：最新提交hash（便于回退）

#### ❌ 不需要记录
1. ~~详细的文件完整路径~~（只需记录文件类型和数量）
2. ~~每个阶段的详细执行过程~~（记录在Git提交信息中）
3. ~~编译错误和修复过程~~（临时信息，不需要保留）
4. ~~中间讨论和试错过程~~（清理掉）

## 标准初始化话术

### 情况1：成功提取 moduleCode

我已创建/更新开发会话状态文件 `dev-session.md`，并提取到以下信息：

- **功能名称**：{功能模块名称}
- **moduleCode**：{实际值}
- **acronym**：{实际值}

请确认这些信息是否正确？
- 若回复 **正确**：我将使用这些信息继续初始化开发环境
- 若回复 **不正确**：请提供正确的信息

### 情况2：未能提取 moduleCode

我已创建/更新开发会话状态文件 `dev-session.md`，但未能自动提取到 `moduleCode` 和 `acronym`。

请提供以下信息：
- **moduleCode** 是什么？
- **acronym** 是什么？
- **功能名称** 是什么？

我将使用这些信息继续初始化开发环境。

## 验证清单

在创建/更新 `dev-session.md` 后，请验证以下内容：

- [ ] 文件已正确创建/更新在项目根目录或模块目录下
- [ ] 会话信息完整（功能名称、moduleCode、acronym）
- [ ] 前后端目录位置已记录
- [ ] 开发方案选择已初始化
- [ ] 前端/后端进度表格已初始化
- [ ] 前序设计产物检查清单已创建
- [ ] 「下次恢复指南」部分已准备

## 后续步骤

完成模块开发初始化后，根据用户选择的开发方案继续执行：

### 选项A：前端开发
1. **前端项目结构创建**：根据 FRONTEND-MODULE-STRUCT-RULES.md 创建前端目录结构
2. **前端代码生成**：按照前端开发进度表格逐步执行各阶段

### 选项B：后端开发
1. **后端项目结构创建**：根据 BACKEND-MODULE-STRUCT-RULES.md 创建后端目录结构
2. **后端代码生成**：按照后端开发进度表格逐步执行各阶段

### 选项C：模块集成测试
1. 执行模块集成测试
2. 验证前后端接口对接

## 新方案优势

- ✅ **单一文件**：所有进度在一个文件中（dev-session.md）
- ✅ **内容精简**：只保留恢复所需的关键信息
- ✅ **快速恢复**：清理上下文后可以立即继续
- ✅ **节省Token**：减少冗余信息，降低token消耗
- ✅ **易于维护**：只需更新一个文件
