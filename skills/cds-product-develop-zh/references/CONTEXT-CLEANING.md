# 上下文清理规范

## 概述

本规范定义了CDS开发阶段执行时的上下文清理策略，目的是减少Token消耗、提高相关信息密度、**避免上下文窗口溢出**。

> ⚠️ **重要：上下文清理是强制执行的，不可跳过！**
> 模型最大上下文限制为 102400 tokens，如果不清理，会导致 API 错误：
> `"This model's maximum context length is 102400 tokens. However, you requested XXX tokens"`

## 清理原则

1. **强制清理原则**：每个阶段完成后**必须**清理，不得跳过
2. **保留关键结论**：将核心结果、关键决策、用户确认写入进度文件
3. **清理中间过程**：移除调试信息、试错过程、临时文件路径
4. **压缩历史信息**：将详细执行过程压缩为执行摘要
5. **按需加载**：详细内容存放在references目录，仅在需要时读取
6. **上下文监控**：实时监控 token 使用量，超过阈值立即清理

## 进度文件格式

**进度文件**：`dev-session.md`（统一跟踪文件）

> 💡 **说明**：已合并 dev-task.md、frontend-code-gen-tracker.md、backend-code-gen-tracker.md 到统一的 dev-session.md 中，减少文件冗余，降低维护成本。
> 
> **详细格式规范请参考**: [DEV-SESSION-FORMAT.md](DEV-SESSION-FORMAT.md)

**保留内容**：
- 会话信息（功能名称、moduleCode、acronym）
- 前后端代码目录路径
- 前端/后端开发进度（阶段状态表）
- 下一阶段预告和需要读取的指南
- 关键文件路径（前序设计文件 + 已生成文件类型和数量）
- Git 提交记录（最新commit hash）
- 关键开发决策
- 下次恢复指南

**清理内容**：
- 代码生成过程
- 编译错误和修复过程
- 调试信息
- 中间讨论和试错过程
- 详细的代码实现细节
- ~~详细的文件完整路径~~（只记录文件类型和数量）

## /clear 指令使用规则

### 使用时机（满足任一条件必须清理）
1. **阶段完成后**（前端7个阶段/后端9阶段的任意一个完成） - **强制执行**
2. 对话历史超过 **30轮** - **强制执行**
3. 上下文接近 token 限制（**>80000 tokens**） - **强制执行**
4. 切换到新模块时
5. 用户明确要求时

### ⚠️ 清理前准备（必须严格执行 - 7项检查）

**清理检查清单**：
- [ ] ✅ 所有关键结论已写入进度文件（dev-session.md）
- [ ] ✅ 前端/后端进度已更新（阶段状态表）
- [ ] ✅ 所有产物已保存到正确位置
- [ ] ✅ 下次任务需要的文件名称已记录（前序设计文件 + 本次生成文件）
- [ ] ✅ 恢复上下文的关键信息已记录（下次恢复指南部分）
- [ ] ✅ 下一步操作已明确
- [ ] ✅ Git 提交记录已更新（最新commit hash）

**只有7项检查全部通过后，才能执行 `/clear` 指令**

> 💡 **注意**：Git 提交是开发流程的独立步骤，**不是清理上下文的前置条件**。建议在阶段完成后尽快提交，但如果 token 即将超限，可以先清理上下文再提交。

## 关键文件路径记录规范（防止清理后丢失）

### 必须在进度文件中记录的文件清单

**前端代码生成**（dev-session.md）：
```markdown
### 下次任务需要的关键文件
- **前序设计文件**：
  - 需求详细设计：`{功能名称}-需求详细设计.md`
  - 前端架构设计：`{功能名称}-前端架构设计.md`
  - API设计文档：`{功能名称}-API设计文档.md`
  - UI原型文件：`{功能名称}.pen`
- **本次生成的文件**：
  - 类型定义：`src/types/{模块名}.ts`
  - API服务：`src/api/modules/{模块名}.ts`
  - 组件文件：`src/components/{组件名}/index.tsx`
  - 页面文件：`src/pages/{模块名}/{页面名}.tsx`
- **下次任务起点**：阶段X+1 - {阶段名称}
```

**后端代码生成**（dev-session.md）：
```markdown
### 下次任务需要的关键文件
- **前序设计文件**：
  - 需求详细设计：`{功能名称}-需求详细设计.md`
  - 后端架构设计：`{功能名称}-后端架构设计.md`
  - API设计文档：`{功能名称}-API设计文档.md`
- **本次生成的文件**：
  - Entity类：`Custom{ModuleCode}{ModelCode}Entity.java`
  - Controller类：`Custom{ModuleCode}{ModelCode}Controller.java`
  - Service类：`Custom{ModuleCode}{ModelCode}Service.java`
  - Dao类：`Custom{ModuleCode}{ModelCode}Dao.java`
- **下次任务起点**：阶段X+1 - {阶段名称}
```

### 记录规则
1. **绝对路径或相对路径**：必须记录完整的文件路径（相对于项目根目录）
2. **文件存在性确认**：记录前确认文件确实存在
3. **下次任务起点**：明确记录下次从哪个阶段/步骤开始
4. **关键标识**：moduleCode、acronym 必须记录

## 恢复上下文信息规范（确保清理后可继续）

### 必须保留的恢复信息

在进度文件（dev-session.md）中必须包含以下信息：

```markdown
## 当前开发状态（恢复用）

### 基本信息
- **功能名称**：{功能名称}
- **moduleCode**：{实际值}
- **acronym**：{实际值}
- **当前开发类型**：[前端 / 后端]

### 进度信息
- **已完成阶段**：阶段1~X
- **当前阶段**：阶段X+1 - {阶段名称}
- **当前阶段状态**：[未开始 / 进行中]
- **阶段完成时间**：{YYYY-MM-DD HH:mm:ss}

### 文件清单（下次任务需要）
- **前序设计文件**：
  - [ ] `{功能名称}-需求详细设计.md` - 路径：`{相对路径}`
  - [ ] `{功能名称}-前端/后端架构设计.md` - 路径：`{相对路径}`
  - [ ] `{功能名称}-API设计文档.md` - 路径：`{相对路径}`
- **本次已生成文件**：
  - [ ] 文件1 - 路径：`{相对路径}`
  - [ ] 文件2 - 路径：`{相对路径}`
  - [ ] ...（列出所有本次生成的文件）

### 下次任务起点
- **从哪个阶段开始**：阶段X+1 - {阶段名称}
- **需要读取的指南文件**：`{指南文件名}.md`
- **需要检查的前置产物**：[文件列表]

### 关键决策（影响后续开发）
- 决策1：{描述}
- 决策2：{描述}

### Git 提交记录
- 最新提交：`{commit hash}` - {提交信息}
```

### 恢复流程
清理后，下次任务开始时：
1. 读取 dev-session.md
2. 从「当前开发状态」获取恢复信息
3. 根据「文件清单」定位需要的文件
4. 从「下次任务起点」确定从哪里继续
5. 加载对应的指南文件，继续开发

## 前端代码生成清理场景

### 清理时机
**每个前端阶段（1-7）完成后立即执行清理** - 不得延后或跳过

### 清理清单
**保留到进度文件**：
- 阶段X完成记录（X=1~7）
- 生成的文件清单（如：src/types/*.ts, src/api/modules/*.ts）
- moduleCode确认记录
- 关键决策（如：API路径选择、组件拆分方案）
- Git commit hash
- 下一阶段预告

**从对话中清理**：
- 类型定义的详细字段讨论
- API接口的调试过程
- 组件实现的试错过程
- 样式调整的中间版本
- 国际化文案的反复修改

### 清理后状态
对话历史应压缩为：
```markdown
### 阶段X完成摘要
- 生成文件：[文件列表]
- 关键决策：[决策记录]
- Git提交：[commit hash]
- 下一步：阶段X+1 - {阶段名称}
```

### ⚠️ 清理执行流程
1. 完成阶段X的代码生成
2. 更新 dev-session.md（前端开发进度部分）
3. 更新 dev-session.md（下次恢复指南部分）
4. **记录下次任务需要的文件清单**（关键！）
5. **向用户展示清理清单**（保留内容 vs 清理内容）
6. 获得用户确认
7. **执行 `/clear` 指令**
8. 对话历史压缩为阶段X完成摘要

> 💡 **Git 提交说明**：建议在阶段完成后尽快执行 `git commit`，但如果 token 即将超限（>80000），**可以先清理上下文再提交 Git**。Git 提交记录会保存在进度文件中，清理后不会丢失。

## 上下文监控机制

### Token 使用量监控
- **安全阈值**：< 60000 tokens - 正常工作
- **警告阈值**：60000-80000 tokens - 准备清理
- **危险阈值**：> 80000 tokens - **必须立即清理**
- **超限错误**：> 102400 tokens - API 报错，任务失败

### 对话轮数监控
- **安全阈值**：< 20 轮 - 正常工作
- **警告阈值**：20-30 轮 - 准备清理
- **危险阈值**：> 30 轮 - **必须立即清理**

### 自动清理触发条件
满足以下任一条件，**必须立即执行清理**：
1. 当前阶段代码生成完成
2. 对话轮数 > 30 轮
3. 上下文接近 80000 tokens
4. 用户明确要求清理

## 最佳实践

1. **及时清理**：每个步骤完成后立即清理，不要累积
2. **完整记录**：清理前确保所有关键信息已保存到进度文件和跟踪文件
3. **按需加载**：详细内容放在references目录，仅在需要时读取
4. **用户透明**：清理过程对用户透明，用户只看到关键结论和下一步操作
5. **单一文件**：所有进度信息统一写入 dev-session.md
6. **清理确认**：执行清理前向用户展示将要保留和清理的内容清单
7. **⚠️ 强制执行**：不得因任何理由跳过清理步骤，否则会导致 token 溢出错误
