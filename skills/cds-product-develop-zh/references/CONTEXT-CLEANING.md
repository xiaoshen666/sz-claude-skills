# 上下文清理规范

## 概述

本规范定义了CDS开发阶段执行时的上下文清理策略，目的是减少Token消耗、提高相关信息密度、避免上下文窗口溢出。

## 清理原则

1. **保留关键结论**：将核心结果、关键决策、用户确认写入进度文件
2. **清理中间过程**：移除调试信息、试错过程、临时文件路径
3. **压缩历史信息**：将详细执行过程压缩为执行摘要
4. **按需加载**：详细内容存放在references目录，仅在需要时读取

## 进度文件格式

**进度文件**：dev-session.md、dev-task.md、frontend-code-gen-tracker.md、backend-code-gen-tracker.md

**保留内容**：
- 当前开发阶段和步骤（前端/后端）
- 前后端代码目录路径
- moduleCode和acronym
- Git 提交记录
- 关键开发决策
- 用户确认记录
- 生成的文件清单
- 下一阶段预告

**清理内容**：
- 代码生成过程
- 编译错误和修复过程
- 调试信息
- 中间讨论和试错过程
- 详细的代码实现细节
- 临时文件路径

## /clear 指令使用规则

### 使用时机
1. **阶段完成后**（前端7个阶段/后端9个阶段的任意一个完成）
2. 对话历史超过50轮
3. 切换到新模块时
4. 用户明确要求时

### 清理前准备（必须严格执行）
1. ✅ 所有关键结论已写入进度文件（dev-task.md / dev-session.md）
2. ✅ 阶段跟踪文件已更新（frontend-code-gen-tracker.md / backend-code-gen-tracker.md）
3. ✅ 所有产物已保存到正确位置
4. ✅ 所有用户确认已记录
5. ✅ 下一步操作已明确
6. ✅ Git 提交已完成

## 前端代码生成清理场景

### 清理时机
每个前端阶段（1-7）完成后立即执行清理

### 清理清单
**保留到进度文件**：
- 阶段X完成记录（X=1~7）
- 生成的文件清单（如：src/types/*.ts, src/api/modules/*.ts）
- moduleCode确认记录
- 关键决策（如：API路径选择、组件拆分方案）
- Git commit hash
- 下一阶段预告

**从对话中清理**：
- 类型定义的详细字段讨论
- API接口的调试过程
- 组件实现的试错过程
- 样式调整的中间版本
- 国际化文案的反复修改

### 清理后状态
对话历史应压缩为：
```
阶段X完成摘要：
- 生成文件：[文件列表]
- 关键决策：[决策记录]
- Git提交：[commit hash]
- 下一步：阶段X+1 - {阶段名称}
```

## 后端代码生成清理场景

### 清理时机
每个后端阶段（1-9）完成后立即执行清理

### 清理清单
**保留到进度文件**：
- 阶段X完成记录（X=1~9）
- 生成的Java类清单（如：Entity、Controller、Service）
- moduleCode和acronym确认记录
- 数据库类型选择记录
- 关键决策（如：ORM方案、事务策略）
- Git commit hash
- 下一阶段预告

**从对话中清理**：
- Entity字段映射的详细讨论
- Controller接口的调试过程
- Service实现的试错过程
- SQL语法的反复修改
- 配置文件的调整过程

### 清理后状态
对话历史应压缩为：
```
阶段X完成摘要：
- 生成文件：[文件列表]
- 关键决策：[决策记录]
- Git提交：[commit hash]
- 下一步：阶段X+1 - {阶段名称}
```

## 最佳实践

1. **及时清理**：每个步骤完成后立即清理，不要累积
2. **完整记录**：清理前确保所有关键信息已保存到进度文件和跟踪文件
3. **按需加载**：详细内容放在references目录，仅在需要时读取
4. **用户透明**：清理过程对用户透明，用户只看到关键结论和下一步操作
5. **双重备份**：关键信息同时写入进度文件（dev-task.md）和跟踪文件（frontend-code-gen-tracker.md / backend-code-gen-tracker.md）
6. **清理确认**：执行清理前向用户展示将要保留和清理的内容清单
