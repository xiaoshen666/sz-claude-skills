# 开发会话跟踪文件

> 💡 **设计原则**：一个文件跟踪所有进度，内容精简，避免冗余

## 文件职责

**dev-session.md** 是唯一的进度跟踪文件，包含：
- 全局会话信息
- 当前模块信息
- 前端开发进度（精简版）
- 后端开发进度（精简版）
- 下次恢复所需的关键信息

## 字段精简原则

### ✅ 必须保留
1. **功能标识**：功能名称、moduleCode、acronym
2. **进度信息**：已完成阶段、下一阶段
3. **恢复信息**：需要读取的指南、前置产物
4. **目录信息**：前后端代码目录路径
5. **Git记录**：最新提交hash（便于回退）

### ❌ 不需要记录
1. ~~详细的文件完整路径~~（只需记录文件类型和数量）
2. ~~每个阶段的详细执行过程~~（记录在Git提交信息中）
3. ~~编译错误和修复过程~~（临时信息，不需要保留）
4. ~~中间讨论和试错过程~~（清理掉）
5. ~~dev-task.md的重复信息~~（合并到dev-session.md）

## 与旧文件的对比

### 旧方案（4个文件）
```
dev-session.md          (全局会话)
dev-task.md             (模块任务) - 与dev-session重复
frontend-code-gen-tracker.md  (前端进度)
backend-code-gen-tracker.md   (后端进度)
```

### 新方案（1个文件）
```
dev-session.md          (统一跟踪)
  ├── 会话信息
  ├── 前端进度（精简版）
  ├── 后端进度（精简版）
  └── 恢复指南
```

### 优势
- ✅ 文件数量：4个 → 1个（减少75%）
- ✅ 内容冗余：消除 dev-session 和 dev-task 的重复
- ✅ 维护成本：只需更新1个文件
- ✅ Token消耗：大幅减少（精简了详细路径和过程记录）
- ✅ 恢复效率：所有信息在一个文件中，快速定位

## 使用规范

### 创建时机
- 开始新模块开发时创建
- 从设计阶段进入开发阶段时创建

### 更新时机
- 每个阶段完成后更新对应进度
- 记录关键决策
- 更新Git提交信息

### 清理上下文时
1. 确保 dev-session.md 已更新
2. 确保「下次恢复指南」部分完整
3. 执行 `/clear` 指令
4. 下次从 dev-session.md 的「下次恢复指南」快速恢复

## 迁移指南

### 从旧方案迁移到新方案

如果您已有旧格式的进度文件：

1. **合并信息**：将 dev-task.md 的信息合并到 dev-session.md
2. **精简内容**：移除详细的文件完整路径，只保留关键信息
3. **统一格式**：按照新格式重新组织前端和后端进度
4. **删除旧文件**：删除 dev-task.md、frontend-code-gen-tracker.md、backend-code-gen-tracker.md
5. **保留备份**：可以先备份旧文件，确认新文件正常后再删除

## 总结

新方案的核心优势：
- 🎯 **单一文件**：所有进度在一个文件中
- 📦 **内容精简**：只保留恢复所需的关键信息
- 🔄 **快速恢复**：清理上下文后可以立即继续
- 💰 **节省Token**：减少冗余信息，降低token消耗
- 📝 **易于维护**：只需更新一个文件
