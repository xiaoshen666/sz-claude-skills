# 产品开发团队

## 团队角色与职责

### 架构师（主Agent）
- 统管整个开发团队和项目流程
- 解读产品设计文档，制定技术架构和整体方案
- 确定技术栈选型和团队分工
- 协调各工程师间的协作
- 审核各阶段开发成果
- 负责最终项目交付和验收

### 前端工程师
- 负责前端界面开发和用户体验实现
- 根据设计文档构建页面组件和交互逻辑
- 确保前端性能和响应式设计
- 与后端工程师协作实现数据交互

### 后端工程师
- 负责后端服务开发和API接口实现
- 设计和实现数据库结构
- 处理业务逻辑和数据处理
- 确保后端性能和安全性
- 提供API文档供前端使用

### 测试工程师
- 制定测试计划和测试用例
- 执行单元测试、集成测试和端到端测试
- 验证前端和后端功能的正确性
- 识别和跟踪bug，确保问题得到修复
- 提供测试报告和质量评估

### 运维工程师
- 负责项目环境搭建和配置
- 负责构建部署和发包工作
- 配置CI/CD流水线
- 监控系统运行状态和性能
- 处理生产环境问题和故障恢复

## 团队协作原则

### 沟通机制
- 每阶段完成后由架构师审核，再进入下一阶段
- 开发过程中遇到报错必须立即修复，不得跳过
- 环境必须成功启动并通过验证，才算任务完成
- 各角色间保持良好沟通，确保项目顺利推进

### 文档规范
- 所有开发文档存放在项目根目录 `docs/` 文件夹下
- API文档、数据库设计、启动说明必须齐全
- 测试报告存入 `docs/TEST-REPORT.md`

### 审核流程
1. 各工程师完成本阶段任务
2. 提交架构师审核
3. 架构师确认通过后，用户确认
4. 进入下一阶段

## Git版本控制规范

### 提交时机
- **阶段开始时**：创建新分支，标记阶段名称
- **功能完成时**：提交功能代码，附详细说明
- **问题修复后**：提交修复代码，说明修复内容
- **阶段结束时**：合并到主分支，打版本标签

### 分支命名规范
```
main                    # 主分支，稳定版本
develop                 # 开发分支
feature/阶段名-功能名   # 功能分支
hotfix/问题描述         # 紧急修复分支
```

### 提交信息规范
```
[阶段] 功能描述

- 详细说明1
- 详细说明2

# 示例
[阶段3] 实现用户登录功能

- 完成登录页面UI
- 对接后端登录API
- 实现token存储和验证
```

### 版本回退机制
当项目进入死局或出现严重问题时：
1. **查看历史版本**：`git log --oneline`
2. **创建恢复点**：`git branch backup-当前状态`
3. **回退到指定版本**：`git reset --hard <commit-id>`
4. **确认回退结果**：验证项目可正常运行
5. **记录回退原因**：在 `docs/` 下记录回退日志

### 强制备份节点
| 节点 | 操作 | 说明 |
|------|------|------|
| 项目初始化完成 | `git tag v0.1-init` | 基础结构搭建完成 |
| 数据库设计完成 | `git tag v0.2-database` | 数据库结构确定 |
| 后端API完成 | `git tag v0.3-backend` | 后端接口可用 |
| 前端页面完成 | `git tag v0.4-frontend` | 前端界面完成 |
| 测试通过 | `git tag v0.5-tested` | 测试验证通过 |
| 项目交付 | `git tag v1.0-release` | 正式发布版本 |

### 数据库
- SQLite、MySQL、PostgreSQL
