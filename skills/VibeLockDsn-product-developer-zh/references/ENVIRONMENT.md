## 二、启动前检查
在启动服务前，先检查以下项目：
- [ ] `.env` 文件存在且配置正确
- [ ] 数据库文件/服务可访问（SQLite文件存在 / MySQL服务运行）
- [ ] 所有依赖已安装（`node_modules/` 存在 / pip包已安装）
- [ ] 端口未被占用（前端默认5173、后端默认8000/3000）

**端口检查与自动更换：**
```bash
lsof -i :5173   # 前端端口
lsof -i :8000   # 后端端口（FastAPI）
lsof -i :3000   # 后端端口（Node.js）
```
- 如端口被占用，自动选择一个闲置端口（如 5174、8001、3001 等，依次递增直到找到可用端口）
- 更换端口后，同步更新 `.env` 中的端口配置和前端 API baseURL，确保前后端端口一致

## 二、启动顺序
**必须按以下顺序启动，每步确认成功后再启动下一个：**

### 1. 启动数据库
- **SQLite**：无需启动服务，确认 `.db` 文件存在即可
- **MySQL**：`mysql.server start` 或 `brew services start mysql`，确认连接成功

### 2. 启动后端
```bash
# Python FastAPI
cd backend && uvicorn app.main:app --reload --port 8000

# Node.js Express
cd backend && npm run dev
```
**验证**：访问 `http://localhost:8000/health`（或对应端口），返回 200

### 3. 启动前端
```bash
cd frontend && npm run dev
```
**验证**：浏览器访问 `http://localhost:5173`，页面正常渲染

## 三、启动失败诊断流程

遇到启动失败时，按以下流程逐步排查：

### 常见错误类型及修复

| 错误类型 | 症状 | 修复方案 |
|---------|------|---------|
| 依赖缺失 | `Module not found` / `ImportError` | 重新安装依赖 |
| 端口占用 | `EADDRINUSE` / `address already in use` | `kill -9 $(lsof -ti:端口号)` |
| 环境变量缺失 | `undefined` / 连接失败 | 检查并补全 `.env` 文件 |
| 数据库连接失败 | `Connection refused` / `SQLITE_CANTOPEN` | 检查数据库服务/文件路径 |
| 迁移未执行 | 表不存在错误 | 执行数据库迁移脚本 |
| Python虚拟环境 | 包找不到 | 激活虚拟环境 `source venv/bin/activate` |
| Node版本不兼容 | 语法错误 / API不存在 | 检查 `.nvmrc`，切换正确版本 |
| CORS错误 | 前端请求被拒绝 | 检查后端CORS配置，添加前端地址 |
| 配置文件错误 | 启动时解析失败 | 检查 `vite.config` / `package.json` 语法 |

### 诊断步骤
1. 完整读取错误日志，定位错误类型
2. 根据上表匹配修复方案
3. 执行修复
4. 重新启动对应服务
5. 验证是否解决，未解决则继续排查

### 修复后必须重新验证
- 后端健康检查接口返回 200
- 前端页面正常渲染
- 前端能成功调用后端 API（无 CORS 错误、无 404）
