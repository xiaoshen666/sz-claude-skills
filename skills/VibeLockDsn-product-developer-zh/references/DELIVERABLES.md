## 四、产物检查规范

### 产物完整性检查
每个步骤完成后，检查该步骤的产物是否完整：
```
1. 列出该步骤应有的所有产物
2. 逐一检查产物是否存在
3. 检查产物内容是否完整
4. 记录缺失或不完整的产物
5. 补充缺失产物后再继续
```

### 产物质量检查
- **项目初始化**：目录结构完整、依赖配置正确
- **数据库设计**：表结构与数据架构一致、外键正确
- **后端服务**：API符合功能清单、业务逻辑符合业务流程
- **前端界面**：页面符合原型设计、UI符合设计规范
- **模块测试**：测试用例覆盖完整、测试通过

### 产物依赖检查
```
设计产物 → 后端开发：模块功能清单、模块业务流程、模块数据架构必须先确认
设计产物 → 前端开发：模块原型、模块设计规范必须先确认
后端开发 → 前端开发：API接口必须先完成
前后端开发 → 模块测试：前后端必须先集成完成
模块测试 → 模块完成：测试必须先通过
```

## 六、Git标签规范

### 标签命名规则
| 标签格式 | 说明 | 示例 |
|---------|------|------|
| v0.1-init | 项目初始化完成 | v0.1-init |
| v0.2-database | 数据库设计完成 | v0.2-database |
| v[X].1-backend | 模块后端完成 | v1.1-backend |
| v[X].2-frontend | 模块前端完成 | v1.2-frontend |
| v[X].3-tested | 模块测试完成 | v1.3-tested |
| v1.0-release | 项目交付 | v1.0-release |

### 标签创建时机
- 每个强制备份节点完成后立即创建
- 创建标签前确保代码已提交
- 标签信息包含简要说明

### 标签使用场景
- 问题回退：回退到上一个稳定版本
- 进度查看：查看各阶段完成情况
- 版本对比：对比不同版本的差异
