# 工作流程规范

## 一、步骤衔接规则

### 全局开发阶段衔接

```
读取设计产物 → 确定技术栈与分工
    衔接内容：设计摘要 → 技术选型
    
确定技术栈 → 项目初始化
    衔接内容：技术栈 → 项目结构、依赖配置
    
项目初始化 → 数据库设计与实现
    衔接内容：数据架构设计 → 数据库表结构
    
数据库设计完成 → 模块开发阶段
    衔接内容：全局基础设施 → 模块开发基础
```

### 模块开发阶段衔接

```
模块初始化 → 后端服务开发
    衔接内容：设计产物 → API设计
    
后端服务开发 → 前端界面开发
    衔接内容：API接口 → 前端调用
    
前端界面开发 → 模块测试
    衔接内容：前后端集成 → 测试验证
    
模块测试 → 模块完成确认
    衔接内容：测试通过 → 更新进度
    
模块完成 → 下一模块
    衔接内容：更新dev-session.md → 选择下一模块
```

### 集成交付阶段衔接

```
所有模块完成 → 全局集成测试
    衔接内容：模块集成 → 端到端测试
    
集成测试通过 → 项目交付
    衔接内容：测试报告 → 交付清单
    
项目交付 → 构建部署
    衔接内容：交付物 → 部署配置
    
构建部署 → 项目验收
    衔接内容：部署完成 → 用户验收
```

## 二、上下文清理机制

### 清理时机
1. **步骤切换时**：从一个步骤进入下一个步骤前
2. **模块切换时**：从一个模块进入下一个模块前
3. **阶段切换时**：从全局开发进入模块开发时

### 清理原则
- **保留**：当前步骤需要的所有设计产物信息
- **保留**：dev-session.md 和 dev-task.md 的当前状态
- **保留**：技术栈选择和项目结构信息
- **清理**：已完成的中间讨论过程
- **清理**：与当前步骤无关的历史决策过程

### 清理操作

#### 步骤切换清理
```
1. 总结当前步骤的关键产出
2. 记录执行摘要到进度文件
3. 清理中间讨论过程
4. 加载下一步骤的 reference 文件
5. 确认下一步骤的前置产物
```

#### 模块切换清理
```
1. 更新 dev-session.md：
   - 当前模块状态 → "已完成"
   - current_module → null
2. 清理当前模块的所有中间产物讨论
3. 保留全局设计产物和技术栈信息
4. 选择下一个模块
5. 执行模块初始化
```

### 上下文保留清单

| 阶段 | 必须保留的上下文 |
|------|-----------------|
| 全局开发 | 设计产物摘要、技术栈选择、项目结构 |
| 模块开发 | 设计产物、技术栈、当前模块的设计文件 |
| 集成交付 | 所有模块的开发成果、测试报告 |

## 三、异常处理机制

### 文件异常

#### dsg-session.md 不存在
```
处理流程：
1. 提示用户："未找到设计进度文件，请先完成产品设计阶段"
2. 停止开发流程，等待用户完成设计
```

#### dev-session.md 不存在
```
处理流程：
1. 检查 dsg-session.md 确认设计进度
2. 如设计已完成，创建新的 dev-session.md
3. 从第一个已设计模块开始开发
```

#### dev-session.md 损坏
```
处理流程：
1. 尝试解析文件内容
2. 如可部分解析，提取有效信息并重建
3. 如完全无法解析，提示用户：
   "开发进度文件损坏，是否重新开始？"
4. 用户确认后，备份损坏文件并创建新文件
```

#### dev-task.md 不存在
```
处理流程：
1. 检查 dev-session.md 中的模块状态
2. 如状态为"进行中"，提示用户：
   "模块任务文件丢失，是否重新开始该模块？"
3. 用户确认后，重置模块状态并重新开始
```

### 开发异常

#### 代码编译失败
```
处理流程：
1. 立即停止当前步骤
2. 分析错误原因
3. 修复错误
4. 重新验证
5. 记录错误和修复过程到执行摘要
```

#### 测试失败
```
处理流程：
1. 分析失败原因
2. 定位问题模块（前端/后端）
3. 相关工程师修复问题
4. 重新执行测试
5. 记录问题和修复过程
```

#### 环境启动失败
```
处理流程：
1. 诊断失败原因（数据库/后端/前端）
2. 检查配置文件
3. 修复配置或代码
4. 重新启动
5. 循环直至成功启动
```

### Git异常

#### 回退到上一个版本
```
处理流程：
1. 确认当前问题无法解决
2. 查看Git历史，找到上一个稳定版本标签
3. 提示用户："是否回退到 [标签名]？"
4. 用户确认后执行回退
5. 从回退点重新开始
```

#### 合并冲突
```
处理流程：
1. 识别冲突文件
2. 分析冲突原因
3. 手动解决冲突
4. 重新测试验证
5. 提交解决后的代码
```

### 用户操作异常

#### 用户中途退出
```
处理流程：
1. 保存当前进度到进度文件
2. 记录当前步骤和状态
3. 下次启动时从保存点恢复
```

#### 用户要求跳过步骤
```
处理流程：
1. 检查该步骤是否为可选步骤
2. 如为可选步骤，询问用户确认后跳过
3. 如为必选步骤，提示用户：
   "该步骤为必选步骤，无法跳过"
4. 记录跳过决策到执行摘要
```

#### 用户要求回退步骤
```
处理流程：
1. 确认回退目标步骤
2. 检查是否有对应的Git标签
3. 提示用户："回退将清除该步骤之后的所有产物，是否继续？"
4. 用户确认后：
   - Git回退到对应标签
   - 更新进度文件状态
   - 从目标步骤重新开始
```

## 四、用户交互规范

### 确认点设置
- **必须确认**：技术栈选择后
- **必须确认**：每个模块完成后
- **必须确认**：全局开发完成后进入模块开发前
- **必须确认**：项目交付前
- **可选确认**：每个步骤完成后（根据步骤重要性）

### 确认格式
```
## 请确认 [步骤名称] 产物

### 产物清单
- [ ] 产物1：[描述]
- [ ] 产物2：[描述]

### Git标签
- 标签：[标签名]
- 说明：[标签说明]

### 下一步操作
[下一步骤描述]

请确认是否继续？
选项：
1. 确认并继续
2. 需要修改
3. 查看详细产物
4. 回退到上一版本
```

### 选项提供
- 每次确认提供 2-4 个选项
- 选项必须互斥且完整
- 默认选项标记为"（推荐）"
