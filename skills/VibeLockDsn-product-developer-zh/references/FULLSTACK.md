# 全栈开发指南

## 开发原则
- 严格按照 **数据库 → 后端 → 前端** 顺序开发
- 每层完成后必须验证通过，才能进入下一层
- 遇到报错立即修复，不得带错误进入下一层
- 前后端接口保持一致，字段命名统一
- 每层完成后必须提交git并打标签
- **必须遵循设计产物**：各工程师必须读取并遵循对应的设计产物

---

## 一、数据库层（后端工程师）

### 前置：读取设计产物
开发前必须读取以下产物：

| 产物 | 文件 | 必须遵循 |
|------|------|---------|
| 模块数据架构 | [模块名]/数据架构.md | ✅ 按照实体字段表创建数据库表结构 |
| 功能架构 | 02-功能架构.md | 理解功能模块，确定数据边界 |
| 模块业务流程 | [模块名]/业务流程.md | 理解流程节点，设计数据流转 |

### 开发内容
- 根据设计产物中的数据架构，创建表结构和字段
- 设置主键、外键、索引、唯一约束
- 编写迁移脚本（Alembic / 手写SQL）
- 生成种子数据和演示数据

### 验证检查点 ✓
- [ ] 所有表创建成功，无报错
- [ ] 外键关系正确
- [ ] 种子数据导入成功
- [ ] 执行基础 CRUD 查询返回预期结果
- [ ] 数据结构与 [模块名]/数据架构.md 一致

### Git提交
```bash
git add .
git commit -m "[阶段3] 数据库设计完成

- 创建数据库表结构
- 设置外键和索引
- 导入种子数据"
git tag v0.2-database
```

## 二、后端层（后端工程师）

### 前置：读取设计产物
开发前必须读取以下产物：

| 产物 | 文件 | 必须遵循 |
|------|------|---------|
| 模块功能清单 | [模块名]/功能清单.md | ✅ 根据页面核心操作设计API接口 |
| 模块业务流程 | [模块名]/业务流程.md | ✅ 理解业务逻辑和流程节点，设计API调用顺序 |
| 模块数据架构 | [模块名]/数据架构.md | ✅ 按照实体关系设计数据交互 |
| 功能架构 | 02-功能架构.md | 理解功能模块划分，确定API边界 |
| 模块原型 | [模块名]/原型.pen | 了解页面布局和数据展示需求 |
| 模块PRD | [模块名]/PRD.md | 获取API数据结构和技术实现要求 |

### 开发内容
- 创建 RESTful API 接口（对应功能清单中的每个核心操作）
- 实现业务逻辑层（Service）
- 配置认证授权（JWT，如需）
- 统一错误处理和响应格式
- 编写健康检查接口 `GET /health`

### API 设计规范
- 接口命名遵循功能清单中的页面操作
- 数据结构遵循数据架构中的实体定义
- 业务逻辑遵循业务流程中的流程节点

### API 文档同步
每完成一个接口，同步更新 `docs/API.md`，记录：
- 接口路径、方法、请求参数、响应格式
- 对应的功能清单页面和操作

### 验证检查点 ✓
- [ ] 服务启动无报错
- [ ] `GET /health` 返回 200
- [ ] 所有 CRUD 接口可通过 curl 或工具调用
- [ ] 错误情况返回正确的错误码和信息
- [ ] 认证接口正常工作（如有）
- [ ] API接口覆盖功能清单中的所有核心操作

### Git提交
```bash
git add .
git commit -m "[阶段3] 后端服务开发完成

- 实现API接口
- 编写业务逻辑
- 配置认证授权
- 更新API文档"
git tag v0.3-backend
```

## 三、前端层（前端工程师）

### 前置：读取设计产物
开发前必须读取以下产物：

| 产物 | 文件 | 必须遵循 |
|------|------|---------|
| 模块原型 | [模块名]/原型.pen | ✅ 严格按照原型设计实现页面布局和交互 |
| 模块设计规范 | [模块名]/设计规范.md | ✅ 按照设计规范实现UI（颜色/字体/间距/组件） |
| 设计风格说明 | design-libraries-readme.md | ✅ 按照风格说明文档实现设计风格 |
| 设计组件库 | .lib.pen | ✅ 参考组件库中的组件样式和规范 |
| 模块功能清单 | [模块名]/功能清单.md | ✅ 按照页面清单开发，实现核心操作 |
| 模块业务流程 | [模块名]/业务流程.md | 理解业务流程，实现页面跳转逻辑 |
| 模块交互流程 | [模块名]/交互流程.md | ✅ 按照状态流图实现交互逻辑（如有） |
| 模块数据架构 | [模块名]/数据架构.md | 了解数据结构，设计前端数据模型 |

### 开发内容
- 按功能清单逐页面开发，每个页面对应一个路由
- **严格按照原型设计**实现页面布局（[模块名]/原型.pen）
- **严格按照设计规范**实现UI（颜色/字体/间距/组件）
- **参考设计组件库**（.lib.pen）中的组件样式
- **遵循设计风格说明**（design-libraries-readme.md）中的设计风格
- 集成后端 API（统一封装请求层）
- 实现表单验证、加载状态、错误提示
- 如需动画效果，使用 GSAP / Motion One / Lottie

### 开发顺序建议
1. **读取原型设计**：使用 Pencil MCP 读取 .pen 文件，理解页面布局
2. **读取设计风格说明**：读取 design-libraries-readme.md 了解设计风格
3. **参考设计组件库**：查看 .lib.pen 中的组件样式
4. 搭建路由结构和页面骨架
5. 实现 API 请求层（统一 baseURL、拦截器、错误处理）
6. 逐页面开发（先静态布局，再接入数据）
7. 实现全局状态（用户信息、权限等）

### 页面开发规范
- 页面布局必须与原型设计一致
- 颜色、字体、间距必须遵循设计规范
- 设计风格必须遵循 design-libraries-readme.md
- 组件样式参考 .lib.pen 中的组件
- 交互逻辑必须遵循交互流程设计（如有）
- 核心操作必须覆盖功能清单中的所有操作

### 验证检查点 ✓
- [ ] 所有页面可正常渲染，无控制台报错
- [ ] 页面布局与原型设计一致
- [ ] UI风格与设计规范一致
- [ ] API 请求成功，数据正确展示
- [ ] 表单提交、数据操作功能正常
- [ ] 响应式布局在常见屏幕尺寸下正常显示
- [ ] 加载状态和错误提示正常工作
- [ ] 页面跳转逻辑符合业务流程

### Git提交
```bash
git add .
git commit -m "[阶段3] 前端界面开发完成

- 实现页面组件
- 集成后端API
- 实现交互逻辑"
```

## 全栈联调验证
完成三层开发后，执行端到端联调：
1. 前端发起请求 → 后端处理 → 数据库读写 → 返回前端展示
2. 覆盖功能清单中的每个核心操作
3. 验证数据流转全链路无异常
4. 验证页面交互符合原型设计和业务流程
