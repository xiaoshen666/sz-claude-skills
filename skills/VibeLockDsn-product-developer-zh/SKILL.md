## 启动恢复流程（最高优先级）

每次进入时**必须首先执行**以下流程：

### 步骤0：读取进度文件
```
IF 项目根目录存在 dsg-session.md THEN
    读取 dsg-session.md（设计进度）
    IF 项目根目录存在 dev-session.md THEN
        读取 dev-session.md（开发进度）
        执行对应的恢复操作
    ELSE
        创建 dev-session.md，从第一个已设计模块开始开发
    END IF
ELSE
    提示用户：请先完成产品设计阶段
END IF
```

### 恢复操作决策表
| dsg-session.md状态 | dev-session.md状态 | current_module | 恢复操作 |
|-------------------|-------------------|----------------|---------|
| 设计未完成 | - | - | 提示用户先完成设计阶段 |
| 设计已完成 | 文件不存在 | - | 创建dev-session.md，从第一个已设计模块开始 |
| 设计已完成 | 存在 | 有值且状态为"进行中" | 加载该模块，读取设计产物继续 |
| 设计已完成 | 存在 | 无值或状态非"进行中" | 选择下一个已完成设计的模块 |
| 所有模块已开发 | - | - | 进入集成与交付阶段 |

### 恢复提示格式
```

### 当前步骤

[具体操作描述]

是否继续？
```

## 开发流程框架

### 核心原则
- **串行开发原则**：功能模块只能一个一个进行开发，不可同时展开
- **产物导向原则**：每步只关注当前步骤的交付物
- **上下文清理原则**：开始新步骤前，清理与当前步骤无关的历史信息
- **Git备份原则**：每个阶段/模块完成后必须提交git并打标签

## 步骤执行规范

### 每个步骤执行前
1. **清理上下文**：移除与当前步骤无关的历史对话（参考 WORKFLOW.md）
2. **加载指南**：读取对应的 reference 文件
3. **确认前置产物**：检查前置步骤的交付物是否存在（参考 DELIVERABLES.md）

### 每个步骤执行后
1. **更新进度文件**：立即更新 dev-session.md 或 dev-task.md
2. **记录执行摘要**：记录关键决策、变更、用户确认
3. **用户确认**：展示产物，获取用户确认后继续
4. **产物检查**：确认当前步骤产物完整（参考 DELIVERABLES.md）

### 模块完成确认
1. 更新 dev-session.md 中的模块状态为"已完成"
2. 更新 dev-session.md 中的 current_module 为 null
3. 询问用户是否继续下一个模块
4. 如继续，选择下一个已完成设计的模块并执行模块初始化

## Git版本控制规则

### 强制备份节点
- 项目初始化完成 → v0.1-init
- 数据库设计完成 → v0.2-database
- 模块后端完成 → v[X].1-backend
- 模块前端完成 → v[X].2-frontend
- 模块测试完成 → v[X].3-tested
- 项目交付 → v1.0-release

### 问题回退机制
- 项目进入死局时，通过git回退到上一个稳定版本
- 详细规范：参考 TEAM.md
