# 工作流程规范

## 一、步骤衔接规则

### 全局设计阶段衔接（流程框架梳理）

```
需求分析 → 功能架构设计
    衔接内容：需求要点 → 功能模块拆解
    
功能架构设计 → 设计组件库准备
    衔接内容：设计风格需求 → 组件库生成
    
设计组件库准备 → 全局设计检查
    衔接内容：全局产物 → 检查验收
    
全局设计检查 → 模块设计阶段
    衔接内容：检查通过 → 按优先级逐个设计
```

### 模块设计阶段衔接

```
模块初始化 → 模块功能清单确认
    衔接内容：全局功能架构 → 模块功能清单
    
模块功能清单确认 → 模块数据架构设计
    衔接内容：功能清单 → 数据实体抽象
    
模块数据架构设计 → 模块业务流程设计
    衔接内容：数据实体 → 业务流程编排
    
模块业务流程设计 → 原型绘制
    衔接内容：业务流程 → 页面原型
    
原型绘制 → 原型调整与同步（如需）
    衔接内容：用户反馈 → 原型修改 → 功能清单同步
    
原型调整与同步 → 模块设计检查
    衔接内容：模块产物 → 检查验收
    
模块设计检查 → 交互流程设计（可选）
    衔接内容：检查通过 → 交互状态流
    
原型绘制 + 交互流程 → PRD编写（可选）
    衔接内容：所有设计产物 → PRD文档
    
PRD编写 → 测试用例生成（可选）
    衔接内容：PRD需求 → 测试用例
    
测试用例生成 → PRD测试用例检查
    衔接内容：PRD+测试用例 → 检查验收
    
PRD测试用例检查 → 模块完成
    衔接内容：检查通过 → 更新进度
    
模块完成 → 下一模块
    衔接内容：更新dsg-session.md → 选择下一模块
```

## 二、上下文清理机制

### 清理时机
1. **步骤切换时**：从一个步骤进入下一个步骤前
2. **模块切换时**：从一个模块进入下一个模块前
3. **阶段切换时**：从全局设计进入模块设计时

### 清理原则
- **保留**：当前步骤需要的所有前置产物信息
- **保留**：dsg-session.md 和 dsg-task.md 的当前状态
- **清理**：已完成的中间讨论过程
- **清理**：与当前步骤无关的历史决策过程

### 清理操作

#### 步骤切换清理
```
1. 总结当前步骤的关键产出
2. 记录执行摘要到进度文件
3. 清理中间讨论过程
4. 加载下一步骤的 reference 文件
5. 确认下一步骤的前置产物
```

#### 模块切换清理
```
1. 更新 dsg-session.md：
   - 当前模块状态 → "已完成"
   - current_module → null
2. 清理当前模块的所有中间产物讨论
3. 保留全局设计产物信息
4. 选择下一个模块
5. 执行模块初始化
```

### 上下文保留清单

| 阶段 | 必须保留的上下文 |
|------|-----------------|
| 全局设计 | 用户原始需求、当前步骤的前置产物 |
| 模块设计 | 全局设计产物、当前模块的功能清单和原型 |

## 三、异常处理机制

### 文件异常

#### dsg-session.md 不存在
```
处理流程：
1. 提示用户："未找到设计进度文件，是否开始新项目？"
2. 用户确认后，执行任务初始化步骤
3. 创建新的 dsg-session.md
```

#### dsg-session.md 损坏
```
处理流程：
1. 尝试解析文件内容
2. 如可部分解析，提取有效信息并重建
3. 如完全无法解析，提示用户：
   "进度文件损坏，是否重新开始？"
4. 用户确认后，备份损坏文件并创建新文件
```

#### dsg-task.md 不存在
```
处理流程：
1. 检查 dsg-session.md 中的模块状态
2. 如状态为"进行中"，提示用户：
   "模块任务文件丢失，是否重新开始该模块？"
3. 用户确认后，重置模块状态并重新开始
```

### 状态异常

#### 状态不一致
```
场景：dsg-session.md 显示模块"进行中"，但模块目录不存在

处理流程：
1. 以 dsg-session.md 为准
2. 提示用户："模块目录丢失，是否重新创建？"
3. 用户确认后，创建模块目录和 dsg-task.md
```

#### 步骤状态冲突
```
场景：dsg-task.md 显示步骤"进行中"，但前置步骤未完成

处理流程：
1. 检查前置步骤的交付物是否存在
2. 如不存在，回退到前置步骤
3. 提示用户："前置步骤未完成，需要先完成 [前置步骤名称]"
```

### 用户操作异常

#### 用户中途退出
```
处理流程：
1. 保存当前进度到进度文件
2. 记录当前步骤和状态
3. 下次启动时从保存点恢复
```

#### 用户要求跳过步骤
```
处理流程：
1. 检查该步骤是否为可选步骤
2. 如为可选步骤，询问用户确认后跳过
3. 如为必选步骤，提示用户：
   "该步骤为必选步骤，无法跳过"
4. 记录跳过决策到执行摘要
```

#### 用户要求回退步骤
```
处理流程：
1. 确认回退目标步骤
2. 提示用户："回退将清除该步骤之后的所有产物，是否继续？"
3. 用户确认后：
   - 更新进度文件状态
   - 清除后续步骤的产物
   - 从目标步骤重新开始
```

## 四、用户交互规范

### 确认点设置
- **必须确认**：功能清单输出后、原型设计前
- **必须确认**：每个模块完成后
- **必须确认**：全局设计完成后进入模块设计前
- **可选确认**：每个步骤完成后（根据步骤重要性）

### 确认格式
```
## 请确认 [步骤名称] 产物

### 产物清单
- [ ] 产物1：[描述]
- [ ] 产物2：[描述]

### 关键决策
1. [决策内容]
2. [决策内容]

### 下一步操作
[下一步骤描述]

请确认是否继续？
选项：
1. 确认并继续
2. 需要修改
3. 查看详细产物
```

### 选项提供
- 每次确认提供 2-4 个选项
- 选项必须互斥且完整
- 默认选项标记为"（推荐）"
