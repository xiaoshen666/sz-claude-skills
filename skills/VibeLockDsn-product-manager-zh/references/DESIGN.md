# 模块原型设计指南

## 前置条件
- 模块功能清单已确认
- 模块数据架构设计完成
- 模块业务流程设计完成
- 设计组件库已准备（`.lib.pen`、`design-libraries-readme.md`）

## 必须遵循
- 原型必须基于功能清单绘制
- 原型必须符合数据架构设计
- 原型必须符合业务流程设计
- **必须使用设计组件库**：原型设计必须导入 `.lib.pen` 文件作为组件库
- **必须遵循设计风格文档**：原型设计必须读取 `design-libraries-readme.md` 进行设计

## 设计步骤

### 1. 导入组件库
将 `.lib.pen` 导入 Pencil 作为设计组件库

### 2. 读取风格文档
仔细阅读 `design-libraries-readme.md` 中的设计风格描述

### 3. 基于功能清单绘制原型
确保每个页面都有对应原型

### 4. 使用组件库中的组件
优先使用 `.lib.pen` 中的设计组件

### 5. 页面检查
每完成一个页面，调用 `vibelockdsn-product-tester-zh` 执行原型检查监督

### 6. 用户确认
原型完成后提交用户确认

## 输出格式

### 设计规范文档
- **设计风格方向**：明确的设计风格描述
- **字体规范**：字体选择、字号层级、行高、字重
- **颜色规范**：主色调、辅助色、中性色、文本色
- **布局规范**：栅格系统、间距标准、对齐方式
- **组件规范**：按钮、表单、卡片等组件设计规范
- **动效规范**：过渡效果、动画时长、交互反馈

### 产品原型
- 使用Pencil软件绘制
- 基于设计规范实现视觉效果
- 使用 `.lib.pen` 组件库中的组件
- 遵循 `design-libraries-readme.md` 中的设计风格
- 通过指定MCP通道交付

## 输出文件
- 文件名：`原型.pen`、`设计规范.md`
- 位置：模块子目录

## 原型调整同步流程

### 触发条件
- 用户在Pencil中修改了产品原型
- 系统检测到原型文件发生变化
- 用户确认需要同步更新相关交付物

### 同步步骤

#### 1. 分析原型变更
- 对比修改前后的原型图差异
- 识别新增、删除、修改的页面和功能
- 提取变更的具体内容和影响范围

#### 2. 更新功能清单
- 根据原型变更更新功能清单表
- 添加新增的页面和功能
- 删除不再存在的页面和功能
- 修改变更的页面描述和核心操作

#### 3. 更新相关交付物
- 更新数据架构设计（如新增字段、修改数据结构）
- 更新业务流程（如新增流程节点、修改流程顺序）
- 更新交互流程设计（如新增交互状态、修改交互逻辑）

#### 4. 验证同步结果
- 检查所有交付物的一致性
- 验证更新后的功能清单与原型图匹配
- 确保数据架构与原型中的数据字段一致

#### 5. 生成同步报告
- 记录原型调整的具体内容
- 列出同步更新的所有交付物

# 模块设计检查验收流程

## 触发时机
原型调整与同步完成后

## 执行流程

### 步骤1：检查是否存在检查报告
```
IF [模块名]/模块设计检查报告.md 存在 THEN
    进入步骤3（读取检查报告）
ELSE
    进入步骤2（召唤测试验收助手）
END IF
```

### 步骤2：召唤测试验收助手
提示用户：
```
模块设计已完成，但尚未进行检查验收。
请先调用 vibelockdsn-product-tester-zh 执行模块设计检查。
是否现在召唤测试验收助手？
```

- **用户确认**：调用 `vibelockdsn-product-tester-zh` 执行模块设计检查，等待检查报告生成后返回步骤1
- **用户拒绝**：等待用户完成检查后继续，更新 dsg-task.md 状态为"模块设计待检查"，退出当前流程

### 步骤3：读取检查报告
读取 `[模块名]/模块设计检查报告.md` 内容

### 步骤4：处理检查结果

#### 情况A：检查结论为"通过"
```
展示检查报告摘要给用户确认：
- 设计规范符合性检查结果
- 组件库使用检查结果
- 页面完整性检查结果
- 数据架构一致性检查结果
- 业务流程一致性检查结果

IF 用户确认OK THEN
    在 dsg-task.md 标记"模块设计检查"状态为"已完成"
    继续后续步骤（交互流程设计等）
ELSE
    根据用户反馈进行调整
    调整完成后交由用户确认
    循环直到用户确认OK
END IF
```

#### 情况B：检查结论为"不通过"
```
展示问题清单和遗漏页面清单给用户：
- 设计规范问题
- 组件库使用问题
- 页面遗漏问题
- 数据架构一致性问题
- 业务流程一致性问题

根据问题清单进行调整修改（可能涉及原型、设计规范等）
修改完成后交由用户确认：
"已根据检查报告完成以下修改：
 - [修改项1]
 - [修改项2]
 请确认修改是否满足要求？"

IF 用户确认OK THEN
    调用 vibelockdsn-product-tester-zh 重新执行检查
    等待新检查报告生成后，返回步骤1
ELSE
    根据用户指示继续修改
    修改完成后再次交由用户确认
    循环直到用户确认OK
END IF
```

## 检查验收循环处理原则
1. **问题优先级处理**：严重问题必须修复，中等问题建议修复，轻微问题可酌情处理
2. **修改范围控制**：仅修改检查报告中指出的问题，避免引入新问题
3. **用户确认机制**：每次修改后必须交由用户确认，确保修改方向正确
4. **进度同步更新**：用户确认OK后，立即更新 dsg-task.md 状态
5. **重新检查触发**：修改完成后需重新调用tester进行检查，确保问题已解决

## 输出文件
- 文件名：`模块设计检查报告.md`
- 位置：模块子目录

## 下一步
模块设计检查验收通过后，继续后续步骤（交互流程设计、PRD编写等）。
