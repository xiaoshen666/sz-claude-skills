# 测试用例生成与检查验收指南

## 前置条件
- PRD已编写完成
- 功能清单已确认
- 数据架构已设计
- 业务流程已设计

## 执行流程

### 第一阶段：测试用例生成

#### 步骤1：召唤测试验收助手生成测试用例
调用 `vibelockdsn-product-tester-zh` 执行测试用例生成

#### 步骤2：等待测试用例生成完成
检查 `[模块名]/测试用例.md` 是否存在

---

### 第二阶段：PRD测试用例检查验收

#### 步骤1：检查是否存在检查报告
```
IF [模块名]/PRD测试用例检查报告.md 存在 THEN
    进入步骤3（读取检查报告）
ELSE
    进入步骤2（召唤测试验收助手）
END IF
```

#### 步骤2：召唤测试验收助手
提示用户：
```
PRD和测试用例已完成，但尚未进行检查验收。
请先调用 vibelockdsn-product-tester-zh 执行PRD测试用例检查。
是否现在召唤测试验收助手？
```

- **用户确认**：调用 `vibelockdsn-product-tester-zh` 执行PRD测试用例检查，等待检查报告生成后返回步骤1
- **用户拒绝**：等待用户完成检查后继续，更新 dsg-task.md 状态为"PRD测试用例待检查"，退出当前流程

#### 步骤3：读取检查报告
读取 `[模块名]/PRD测试用例检查报告.md` 内容

#### 步骤4：处理检查结果

##### 情况A：检查结论为"通过"
```
展示检查报告摘要给用户确认：
- PRD覆盖性检查结果
- 测试用例覆盖性检查结果
- 一致性检查结果
- 文档质量检查结果

IF 用户确认OK THEN
    在 dsg-task.md 标记"PRD测试用例检查"状态为"已完成"
    更新 dsg-task.md 模块状态为"已完成"
    更新 dsg-session.md 模块状态为"已完成"
    询问用户是否继续下一个模块
ELSE
    根据用户反馈进行调整
    调整完成后交由用户确认
    循环直到用户确认OK
END IF
```

##### 情况B：检查结论为"不通过"
```
展示问题清单和遗漏功能清单给用户：
- PRD覆盖性问题
- 测试用例覆盖性问题
- 一致性问题
- 文档质量问题

根据问题清单进行调整修改（可能涉及PRD、测试用例等）
修改完成后交由用户确认：
"已根据检查报告完成以下修改：
 - [修改项1]
 - [修改项2]
 请确认修改是否满足要求？"

IF 用户确认OK THEN
    调用 vibelockdsn-product-tester-zh 重新执行检查
    等待新检查报告生成后，返回步骤1
ELSE
    根据用户指示继续修改
    修改完成后再次交由用户确认
    循环直到用户确认OK
END IF
```

---

## 检查验收循环处理原则
1. **问题优先级处理**：严重问题必须修复，中等问题建议修复，轻微问题可酌情处理
2. **修改范围控制**：仅修改检查报告中指出的问题，避免引入新问题
3. **用户确认机制**：每次修改后必须交由用户确认，确保修改方向正确
4. **进度同步更新**：用户确认OK后，立即更新 dsg-task.md 和 dsg-session.md 状态
5. **重新检查触发**：修改完成后需重新调用tester进行检查，确保问题已解决

---

## 输出文件
- 文件名：`测试用例.md`、`PRD测试用例检查报告.md`
- 位置：模块子目录

## 模块完成确认
PRD测试用例检查验收通过后：
1. 更新 dsg-session.md 中的模块状态为"已完成"
2. 更新 dsg-session.md 中的 current_module 为 null
3. 询问用户是否继续下一个模块
4. 如继续，选择下一个"未开始"模块并执行模块初始化
