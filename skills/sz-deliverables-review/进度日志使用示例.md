# 进度日志功能使用示例

## 概述
本文档展示了如何在`sz-deliverables-review`技能中使用进度日志功能来跟踪项目开发进度。

## 基本使用流程

### 1. 首次使用（新项目）
当您开始一个新项目时，`sz-deliverables-review`技能会自动创建进度日志文件。

**示例对话：**
```
用户：请检查"电商平台"项目的需求文档
Claude：好的，我将检查"电商平台"项目的需求文档。

首先，我检查项目根目录是否存在进度日志文件...
检测到这是新项目，正在创建进度日志文件：电商平台-progress-log.json
进度日志文件创建成功！

现在开始检查需求文档...
1. 检查文件夹结构：01需求文档/ ✓
2. 检查文档位置：01需求文档/电商平台需求概要文档.md ✓
3. 检查文档内容：完整性、一致性、准确性...
```

### 2. 进度跟踪
每个环节完成后，进度日志会自动更新。

**进度日志更新示例：**
```json
{
  "project_name": "电商平台",
  "current_phase": "技术架构设计",
  "phases": {
    "需求分析设计": {
      "status": "completed",
      "completed_at": "2024-01-01T10:00:00Z",
      "review_status": "passed",
      "reviewed_at": "2024-01-01T10:30:00Z"
    },
    "技术架构设计": {
      "status": "in_progress",
      "started_at": "2024-01-01T11:00:00Z",
      "review_status": "pending"
    }
  }
}
```

### 3. 恢复工作流程
当您重新启动项目时，技能会自动检测进度并恢复工作流程。

**恢复示例：**
```
用户：请继续"电商平台"项目的工作
Claude：检测到进度日志文件：电商平台-progress-log.json

当前项目状态：
- 项目名称：电商平台
- 当前环节：技术架构设计（进行中）
- 已完成环节：需求分析设计（已通过评审）
- 下一环节：数据库和接口设计

根据进度日志，您需要：
1. 完成技术架构设计环节
2. 调用sz-deliverables-review检查架构设计文档
3. 然后进入数据库和接口设计环节

是否继续技术架构设计环节？
```

## 进度日志操作示例

### 开始新环节
```json
// 开始技术架构设计环节
{
  "action": "phase_started",
  "phase": "技术架构设计",
  "timestamp": "2024-01-01T11:00:00Z",
  "notes": "开始技术架构设计工作"
}
```

### 完成环节
```json
// 完成技术架构设计环节
{
  "action": "phase_completed",
  "phase": "技术架构设计",
  "timestamp": "2024-01-01T15:00:00Z",
  "notes": "技术架构设计文档已完成"
}
```

### 评审通过
```json
// 技术架构设计评审通过
{
  "action": "review_passed",
  "phase": "技术架构设计",
  "timestamp": "2024-01-01T15:30:00Z",
  "notes": "技术架构设计文档质量合格，可以进入下一环节"
}
```

## 恢复策略示例

### 场景1：完全恢复
**情况**：所有前置环节已完成且通过评审
**处理**：直接进入当前环节
```
检测到进度日志，当前环节：前端实现
前置环节状态：
- UI原型设计：已完成 ✓ 已通过评审 ✓
- 数据库和接口设计：已完成 ✓ 已通过评审 ✓
- 技术架构设计：已完成 ✓ 已通过评审 ✓
- 需求分析设计：已完成 ✓ 已通过评审 ✓

可以完全恢复，直接进入前端实现环节。
```

### 场景2：部分恢复
**情况**：某些前置环节未完成
**处理**：需要先完成缺失的环节
```
检测到进度日志，当前环节：后端实现
前置环节状态：
- 前端实现：已完成 ✓ 已通过评审 ✓
- UI原型设计：已完成 ✓ 已通过评审 ✓
- 数据库和接口设计：未完成 ✗
- 技术架构设计：已完成 ✓ 已通过评审 ✓

需要先完成数据库和接口设计环节，然后才能继续后端实现。
```

### 场景3：重新开始
**情况**：进度日志文件损坏
**处理**：创建新的进度日志
```
检测到进度日志文件损坏，无法解析。
是否创建新的进度日志文件并重新开始项目？
```

## 最佳实践

### 1. 定期备份进度日志
```bash
# 备份进度日志文件
cp 项目名称-progress-log.json 项目名称-progress-log-备份.json
```

### 2. 手动修复进度日志
如果进度日志出现问题，可以：
1. 使用模板文件重新创建
2. 手动编辑JSON文件
3. 使用`sz-deliverables-review`技能重新初始化

### 3. 多项目管理
对于多个项目，进度日志文件会自动区分：
- 电商平台-progress-log.json
- 客户管理系统-progress-log.json
- 数据分析平台-progress-log.json

### 4. 团队协作
将进度日志文件纳入版本控制：
```bash
git add 项目名称-progress-log.json
git commit -m "更新项目进度"
```

## 故障排除

### 常见问题
1. **进度日志文件丢失**
   - 解决方案：重新调用`sz-deliverables-review`，会自动创建新的日志文件

2. **进度状态不正确**
   - 解决方案：手动编辑JSON文件或使用技能重新初始化

3. **无法恢复工作流程**
   - 解决方案：检查前置环节状态，确保所有必要环节已完成

4. **JSON格式错误**
   - 解决方案：使用`python3 -m json.tool 文件.json`验证格式

### 获取帮助
如果在使用进度日志功能时遇到问题：
1. 查看`sz-deliverables-review`技能的"进度恢复功能"部分
2. 参考进度日志模板文件的结构
3. 使用测试脚本验证功能
4. 查看使用示例文档

## 总结
进度日志功能为软件开发工作流程提供了：
- **可追溯性**：完整记录项目开发历史
- **可恢复性**：支持项目中断后的继续
- **可视化**：清晰展示项目进度
- **自动化**：自动跟踪和更新状态

通过合理使用进度日志功能，您可以更高效地管理软件开发项目，确保工作流程的连续性和质量。
